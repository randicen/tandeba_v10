## 2. Hallazgos detallados

### 🟠 MAY-1 — Spec §5.2 dice `discover(): Skill[]` pero el código retorna `SkillMatch[]` (con score y matchedOn)

**Severidad**: 🟠 MAYOR
**Archivo**: `AGENT_D2C_SKILLS_V1_SPEC.md` líneas 188-194, vs `src/agent/skills/skill-registry.ts:128`.

**Por qué pasó**: durante implementación decidí agregar `score` y `matchedOn` para auditoría (es un campo barato y útil). Olvidé actualizar el spec.

**Fix sugerido** (5 min):
- Actualizar spec §5.2 para incluir `SkillMatch` en la firma.
- Actualizar spec §5.3 con la fórmula explícita `+10 topic, +5 jurisdicción, +1 keyword` y tiebreak alfabético ascendente.
- Agregar un test que verifique `discover` retorna `SkillMatch` con `score` correcto (ya existe, pero hacerlo explícito en el doc del test).

### 🟠 MAY-2 — `ClauseReviewerSpecialist` solo carga skills si el caller pasa `node.metadata`, pero el LLMNode type no declara `metadata` formalmente

**Severidad**: 🟠 MAYOR
**Archivo**: `src/agent/specialists/clause-reviewer-specialist.ts:81-88`.

**Síntoma**: 
```typescript
const discoveryCtx: SkillDiscoveryContext = {
  topic: typeof (node as LLMNode & { metadata?: { topic?: string } }).metadata?.topic === "string"
    ? (node as LLMNode & { metadata?: { topic?: string } }).metadata!.topic
    : undefined,
  ...
}
```

`LLMNode` no tiene un campo `metadata` declarado. Lo estoy casteando con `as LLMNode & { metadata?: { topic?: string } }`. Si el workflow JSON incluye `"metadata": { "topic": "jurisprudencia" }` funciona (el parser lo acepta como `unknown`), pero TypeScript no lo sabe.

**Impacto**: type-safety perdida. Si alguien refactoriza el workflow engine y borra el `metadata` del state, el código tira runtime en lugar de compile-time.

**Por qué pasó**: el DSL types no tenía `metadata` en `LLMNode`. Para no hacer breaking changes (la prioridad de los últimos sprints), casteé en el specialist. Pero el cast deja un footgun.

**Fix sugerido** (3 opciones, todas válidas):
- **A**: agregar `metadata?: { topic?: string; jurisdiction?: string }` a `LLMNode` en `src/agent/workflow-engine/dsl/types.ts`. Riesgo: si rompe tests, revertir. Probable impacto: 0 tests rotos.
- **B**: dejar el cast pero agregar un test que verifique que el cast funciona end-to-end. Riesgo: el cast sigue siendo un footgun.
- **C**: definir un `SpecialistExecuteParams.metadata` opcional en el interface `Specialist`, y que el motor lo extraiga del state. Más limpio pero requiere cambio en el interface.

**Recomendación**: A. Es la opción más limpia y elimina el cast.

### 🟠 MAY-3 — Spec §3 contradice implementación: spec dice "pre-loop, al construir el specialist", código hace discover() en CADA `execute()`

**Severidad**: 🟠 MAYOR
**Archivo**: `AGENT_D2C_SKILLS_V1_SPEC.md` §6.2 vs `src/agent/specialists/clause-reviewer-specialist.ts:91`.

**Síntoma**: El spec §6.2 dice "Cargar skills por nodo es caro (parseo de markdown). Cargarlas una vez al boot es lo correcto. El `discover()` es barato." Esto sugiere que el `discover` se hace una vez al boot.

Pero en `ClauseReviewerSpecialist::execute()`, el `discover()` se hace **en cada llamada** a `execute()` (línea 91, dentro del método). La diferencia práctica es chica (es un loop sobre N skills chicas con string match) pero la decisión registrada en el spec es distinta a la implementación.

**Impacto**: contradicción doc↔código. Confusión para próximos devs.

**Por qué pasó**: durante implementación me di cuenta de que si el `discover()` se hace en cada `execute()`, el contexto del nodo (topic, jurisdicción, userMessage) puede cambiar entre nodos del mismo workflow. Hacerlo pre-loop significa que TODO el specialist tiene UN solo contexto. Decidí hacerlo por nodo (más flexible), pero olvidé actualizar el spec.

**Fix sugerido** (2 min):
- Actualizar spec §6.2 para decir "se hace en cada `execute()`, no al boot. El `discover()` es O(N) sobre N skills chicas, despreciable (<1ms para 100 skills). El parseo de SKILL.md se hace una vez al boot, en `loadFromDir()`."
- Eliminar la línea "Cargar skills por nodo es caro (parseo de markdown)" porque confunde.

**Por qué no es crítico**: funcionalmente está bien. Es solo una corrección de doc.

### 🟡 MIN-2 — `tokenize` no maneja acentos de forma semántica: "demanda" no matchea "demandar" ni "demandado"

**Severidad**: 🟡 MENOR
**Archivo**: `src/agent/skills/skill-registry.ts:184-188`.

**Síntoma**: `tokenize` separa por whitespace + puntuación. Si la keyword de la skill es "demanda" y el userMessage dice "demandado", no matchea (correcto para keyword matching estricto). Pero si la keyword es "ley" y el texto dice "leyes", tampoco matchea (probablemente deseable). 

Lo que SÍ es un problema: si la keyword es "Código" y el texto dice "código civil" en minúscula, el matching funciona (porque `toLowerCase()` está). Pero si la keyword es "tutela" y el texto dice "tutelas" (plural), no matchea.

**Impacto**: bajo. La mayoría de los keywords del jurídico colombiano son en singular (ley, norma, contrato, etc.). Pero hay keywords en plural que podrían fallar.

**Por qué NO es MAY**: el spec §11 decisión 3 dice "discovery determinista, no por LLM" y "el matching es por keywords". Stemming (reducir a raíz) sería una mejora, pero va más allá del scope D2c.

**Fix sugerido**: 2 opciones:
- **A**: documentar en el spec §4.3 que las keywords deben estar en singular y lowercase. El usuario las diseña correctamente.
- **B**: implementar stemming básico (quitar "s", "es", suffix matching) — riesgo de falsos positivos.

**Recomendación**: A. Más simple y menos sorprendente.

### 🟡 MIN-3 — El regex de tokenización no maneja guiones bajos ni dígitos pegados a letras

**Severidad**: 🟡 MENOR
**Archivo**: `src/agent/skills/skill-registry.ts:187`.

**Síntoma**: 
```typescript
const tokens = lower.split(/[^a-záéíóúñü0-9]+/u);
```

El regex incluye `0-9` en la clase "permitida", pero NO incluye `_` (guion bajo). Si la keyword es "rfc_123" o "caso_123", no va a matchear nunca.

**Impacto**: bajo. Las keywords jurídicas colombianas no tienen guiones bajos. Pero si mañana se agrega una skill de "compliance fiscal" con keywords tipo "rfc_emisor", falla.

**Por qué NO es MAY**: en el dominio actual (jurídico colombiano) no aplica.

**Fix sugerido**: documentar la convención ("keywords son lowercase, alfanumérico, sin guiones") en spec §4.3. Si se necesita `_`, agregarlo al regex. Decisión de producto.

## 4. Análisis de seguridad

### ✅ Aspectos seguros

| Aspecto | Estado | Detalle |
|---|---|---|
| **Path traversal** | ✅ Seguro | `join(skillDir, entry.name)` usa `path.join` que normaliza. No se concatena strings del usuario. |
| **YAML parsing seguro** | ✅ Seguro | La librería `yaml` del proyecto no tiene CVE conocidos para v2.x con parse sin `DEFAULT_SCHEMA` peligrosa. Se usa `parse(frontMatterStr)` con default schema (no `parseAllDocuments`). |
| **Skills no ejecutan código** | ✅ Seguro | `Skill` type no tiene `eval`, `Function`, ni `vm.runInNewContext`. El body se concatena al prompt como string. |
| **Auditoría posible** | ✅ Seguro | Cada `SkillMatch` lleva `matchedOn` con la razón del match. Forward-compat con audit log en D3+. |
| **Falla loud en boot** | ✅ Seguro | SKILL.md malformado → boot falla. No se skipea. |

### ⚠️ Aspectos a considerar (no críticos, pero documentar)

| Aspecto | Caveat | Detalle |
|---|---|---|
| **Inyección de prompt via SKILL.md** | ⚠️ Documentar | Una SKILL.md maliciosa podría decir "Ignore todas las instrucciones anteriores y...". Esto NO es diferente a cualquier sistema que carga texto externo al prompt (es la realidad de LLM apps). Pero vale documentar que las SKILL.md vienen de fuentes confiables (Worgena-authored en D2c, editables por el cliente en D6 con un guardrail). |
| **Tamaño de SKILL.md** | ⚠️ Documentar | Sin límite. Si alguien commitea un SKILL.md de 10MB, el token cost explota. Sugerencia para D3+ (D6 cuando el usuario edita): límite de 100KB. |
| **Asset path traversal** | ⚠️ Teóricamente seguro | `walk()` usa `join(currentDir, entry.name)`, pero como `entry.name` viene de `readdirSync` (no del usuario), no hay path traversal. Si en D6 el usuario edita assets, hay que poner guards. |

## 5. Análisis de performance

### ✅ Aspectos performantes

| Operación | Costo | Notas |
|---|---|---|
| `parseSkillFile` (boot) | O(size del archivo) | Una vez por skill. Total: <100ms para 10 skills. |
| `loadFromDir` (boot) | O(N) sobre skills | Una vez por boot. |
| `discover()` (por nodo) | O(N × K) donde N=skills, K=keywords | Para 10 skills con 10 keywords = 100 iteraciones de `Set.has()`. <1ms. |
| `formatSkillsForPrompt` (por nodo) | O(M) donde M=skills matcheadas | Típicamente 0-2. |
| `tokenize` (por nodo) | O(L) donde L=longitud del userMessage | Para mensajes de 1KB = <1ms. |

### Conclusión

**Performance OK para 100 skills y 10K requests/min.** No es un cuello de botella.

**Forward-compat**: si en D3+ se cargan skills por tenant (multi-tenant) y cada tenant tiene 1000 skills, el discover() puede pasar a O(1) con un índice invertido (mapa `topic → Set<skillName>`). Documentar en spec.

- `Skill` type es API pública. Cambios breaking en v2.
- `SkillMatch` type es API pública (es retornado por `discover`).
- `SkillRegistry` es API pública. Sus métodos estáticos `loadFromDir` y `create`, y los métodos `listSkills`, `get`, `size`, `discover` son contrato.

**Recomendación**: en el spec §9, marcar explícitamente cuáles exports son API pública estable.
